Prompt Vault — a calm room for every prompt you've written.

A single local binary that captures, searches, and serves your prompts from the web, your Codex sessions, and your OpenCode history — with the editorial feel of a well-kept library and the speed of a terminal.

01 — Premise

The shape of it.

The product is one thing: a personal library of prompts you actually use, opened by a keystroke. Everything else serves that one thing.

It runs as a single Go binary. Double-click, and it opens a browser tab pointed at localhost:7777. The whole UI is HTML — embedded in the binary — styled in the Atelier aesthetic. There is no signup, no cloud, no telemetry. Your prompts live in one SQLite file in ~/.prompt-vault/. You can grep it.

Three forces drive every decision:

  • Locality. Your prompts never leave your machine. The binary reads your Codex/OpenCode history directly. The web ingest is a bookmarklet, not a service.
  • Speed. Search must feel instant — under 30 ms from keypress to repaint. SQLite FTS5 on the server, MiniSearch in the browser for typing feedback.
  • Calm. The Atelier design language is non-negotiable: warm paper, restrained motion, Fraunces for type. The tool should feel like a reading room, not a dashboard.
The wedge

No existing tool combines local ingestion of Codex/OpenCode session history with a personal-library UX. Langfuse is for teams, PromptHub charges per seat, Chrome extensions can't read your shell. This sits in the gap.

03 — The stack

Every pick has a reason.

LayerPickWhy this, not the alternative
Language Go 1.22+ Single static binary, trivial cross-compile, stdlib HTTP is enough. Rust/Tauri ships smaller (~5 MB) but the Rust learning tax doesn't pay back here.
HTTP net/http + chi Stdlib for everything; chi only for middleware and path params. No Gin, no Echo — they bring more than we need.
Database modernc.org/sqlite Pure-Go SQLite — no CGO, so cross-compile stays one-shot. ~15% slower than mattn/go-sqlite3 but irrelevant at this scale.
Search (server) SQLite FTS5 BM25 ranking, porter stemming, trigram tokenizer for code-like queries. Built into SQLite. GitHub Copilot CLI uses the same pattern.
Search (client) MiniSearch 7.x ~7 KB gzipped, inverted index, fuzzy + prefix + field boost. Gives instant typing feedback before the server query lands. Fuse.js is slower at scale; FlexSearch's API is fiddlier; Orama is overkill.
File watching fsnotify OS-native watchers for Codex/OpenCode dirs so new sessions appear in the Inbox without polling.
Asset embed go:embed Stdlib. Bundles index.html, CSS, JS, fonts directly into the binary.
Reactivity Alpine.js 3.x ~15 KB, declarative x-data/x-show, no build step. React/Vue would force tooling we don't want.
Type Fraunces + IBM Plex Sans + Mono Self-host the WOFF2 files (one weight each) so the app works offline. Total: ~120 KB.
Icons Lucide (inline SVG) Inline only the ~12 icons we use. No icon-font weight.
Build Makefile + GOOS/GOARCH One make release produces darwin/linux/windows × amd64/arm64.

What we considered and rejected

Go terminal app?
No.
TUIs can't do the calm animations and serif typography that make this product distinct. Copy/paste UX is also worse in a terminal.
Tauri?
Tempting, not now.
~5 MB binary and a native window are real wins, but Rust ownership rules slow the first ship. Revisit once the Go version is stable.
Electron?
No.
100+ MB binary, hundreds of MB RAM idle. Violates the "lightweight" requirement.
Pure browser app (no backend)?
No.
Browsers can't read ~/.codex/sessions. The whole product hinges on local file access.
Vector / semantic search?
Not yet.
FTS5 with porter stemming covers 95% of "find that prompt about refactoring." Add embeddings only if recall becomes a real complaint.
React + Vite?
No.
Build step ruins the "single binary, edit a file, refresh" feedback loop. Alpine.js is the right tool for an app this size.
04 — Data model

One table, one virtual table, three triggers.

SQLite holds everything. The schema is deliberately flat — there is no tags join table because tags are queried as JSON arrays and matched via FTS, not relationally.

internal/store/schema.sqlSQL
-- Core table. id is a ULID so it sorts lexicographically by time.
CREATE TABLE prompts (
  id          TEXT    PRIMARY KEY,
  title       TEXT    NOT NULL,
  body        TEXT    NOT NULL,
  tags        TEXT    NOT NULL DEFAULT '[]',    -- JSON array
  source      TEXT    NOT NULL,                  -- manual|web|codex|opencode
  source_ref  TEXT,                                -- URL, session id, file path
  uses        INTEGER NOT NULL DEFAULT 0,
  pinned      INTEGER NOT NULL DEFAULT 0,
  archived    INTEGER NOT NULL DEFAULT 0,
  body_hash   TEXT    NOT NULL,                  -- SHA-256 prefix for dedup
  created_at  INTEGER NOT NULL,                  -- unix seconds
  updated_at  INTEGER NOT NULL
);
CREATE INDEX idx_prompts_source   ON prompts(source);
CREATE INDEX idx_prompts_updated  ON prompts(updated_at DESC);
CREATE UNIQUE INDEX uq_body_hash  ON prompts(body_hash);

-- Inbox: staged candidates from Codex/OpenCode ingest, awaiting user decision.
CREATE TABLE inbox (
  id          TEXT    PRIMARY KEY,
  body        TEXT    NOT NULL,
  source      TEXT    NOT NULL,
  source_ref  TEXT    NOT NULL,
  body_hash   TEXT    NOT NULL UNIQUE,
  detected_at INTEGER NOT NULL
);

-- FTS5 virtual table — kept in sync via triggers below.
CREATE VIRTUAL TABLE prompts_fts USING fts5(
  title, body, tags,
  content='prompts', content_rowid='rowid',
  tokenize='porter unicode61 remove_diacritics 2'
);

CREATE TRIGGER prompts_ai AFTER INSERT ON prompts BEGIN
  INSERT INTO prompts_fts(rowid, title, body, tags)
  VALUES (new.rowid, new.title, new.body, new.tags);
END;
CREATE TRIGGER prompts_au AFTER UPDATE ON prompts BEGIN
  INSERT INTO prompts_fts(prompts_fts, rowid, title, body, tags)
  VALUES ('delete', old.rowid, old.title, old.body, old.tags);
  INSERT INTO prompts_fts(rowid, title, body, tags)
  VALUES (new.rowid, new.title, new.body, new.tags);
END;
CREATE TRIGGER prompts_ad AFTER DELETE ON prompts BEGIN
  INSERT INTO prompts_fts(prompts_fts, rowid, title, body, tags)
  VALUES ('delete', old.rowid, old.title, old.body, old.tags);
END;

Why this shape

  • ULID for id sorts by creation time without a separate index. Useful for "most recent" without an ORDER BY created_at.
  • body_hash UNIQUE dedupes silently. If you ingest the same Codex prompt twice across two scans, only one row exists.
  • tags as JSON not a join. Faceted filtering happens client-side on the loaded set; full-text covers tag search via FTS.
  • porter + unicode61 + remove_diacritics handles English stemming, mixed scripts, and accented input. For polyglot users add the trigram tokenizer later.
05 — Search

Two layers, one feeling: instant.

The illusion of instantaneous search comes from doing two things at once. The browser shows fuzzy matches the moment you type, while the server runs the proper BM25 query in the background and reconciles when it lands.

Layer 1 — client (MiniSearch)

On load, the app fetches the most-recent 500 prompts (GET /api/prompts?limit=500) and builds a MiniSearch index. Every keystroke runs against that index — under 5 ms typical. The index lives in memory; no IndexedDB.

web/app.js · client indexJS
import MiniSearch from './vendor/minisearch.min.js';

const idx = new MiniSearch({
  fields: ['title', 'body', 'tags'],
  storeFields: ['id', 'title', 'source', 'uses'],
  searchOptions: {
    boost: { title: 3, tags: 2 },
    fuzzy: 0.18,           // tolerate 1 typo per 6 chars
    prefix: true,           // "refac" matches "refactor"
    combineWith: 'AND',
  },
});
idx.addAll(await fetchInitial());

Layer 2 — server (SQLite FTS5)

200 ms after the user stops typing, the client also calls /api/search?q=…. The server runs an FTS5 query with BM25 ranking, then merges any new IDs into the displayed result set. The merge is gentle — already-visible items don't reorder, new items slot in with a soft fade.

internal/store/search.goGo
func (s *Store) Search(ctx context.Context, q string, src string) ([]Prompt, error) {
  const sql = `
    SELECT p.id, p.title, p.body, p.tags, p.source, p.uses, p.pinned
    FROM prompts_fts f
    JOIN prompts p ON p.rowid = f.rowid
    WHERE prompts_fts MATCH ?
      AND (? = '' OR p.source = ?)
      AND p.archived = 0
    ORDER BY bm25(prompts_fts, 3.0, 1.0, 2.0), p.uses DESC
    LIMIT 50;
  `
  return s.query(ctx, sql, ftsQuoteQuery(q), src, src)
}

ftsQuoteQuery wraps each token in double quotes and appends * so partial words match, then ORs them. This avoids FTS5's syntax-error surprises when a user types punctuation.

Performance budget

Cold load < 80 ms · keystroke → first paint < 16 ms (client) · keystroke → server-ranked results < 60 ms · 10 000 prompts indexed in < 600 ms at startup.

06 — Ingestion

Four paths in, one inbox out.

Nothing auto-imports. Every external prompt lands in an Inbox view first, where you keep or discard with a single keystroke. The library stays curated by default.

Codex

Walks $CODEX_HOME/sessions/YYYY/MM/DD/rollout-*.jsonl (default ~/.codex). Each line is a JSON event; we keep only the messages where role == "user" and content is non-empty. The first user prompt of each session is what people usually want — but later prompts in the same session are also offered, lower in the ranking.

internal/ingest/codex.goGo
func ScanCodex(root string) ([]Candidate, error) {
  var out []Candidate
  return out, filepath.WalkDir(root, func(p string, d fs.DirEntry, err error) error {
    if err != nil || d.IsDir() || !strings.HasSuffix(p, ".jsonl") {
      return err
    }
    f, err := os.Open(p)
    if err != nil { return err }
    defer f.Close()
    sc := bufio.NewScanner(f)
    sc.Buffer(make([]byte, 1024*1024), 8*1024*1024)
    for sc.Scan() {
      var ev struct {
        Type     string `json:"type"`
        Payload  struct {
          Role    string `json:"role"`
          Content string `json:"content"`
        } `json:"payload"`
      }
      if err := json.Unmarshal(sc.Bytes(), &ev); err != nil { continue }
      if ev.Payload.Role == "user" && len(ev.Payload.Content) > 20 {
        out = append(out, Candidate{
          Body: ev.Payload.Content, Source: "codex", SourceRef: p,
        })
      }
    }
    return sc.Err()
  })
}

OpenCode

Walks $OPENCODE_DATA_DIR/storage/message/{sessionID}/msg_*.json (default ~/.local/share/opencode). Each file is one message; filter to role == "user". Same dedup-by-hash flow.

Web

A 12-line bookmarklet: select text on any page, click the bookmark, it POSTs { title: document.title, body: selection, url: location.href } to http://localhost:7777/api/ingest/web. CORS is locked to * on that route only — safe because the server binds to 127.0.0.1 by default.

Manual

"+ New" button opens a paper-feeling form (textarea in Fraunces italic, tag chips below). Saves directly to prompts with source = 'manual'.

Dedup & watch

Every candidate gets a SHA-256 of its trimmed body; the first 16 chars become body_hash. The UNIQUE index in SQL is the actual dedup mechanism — Go just inserts and ignores constraint violations.

An fsnotify watcher on the Codex and OpenCode dirs auto-runs ingest on new sessions. A small dot appears next to the Inbox count when new candidates arrive — no notification, no popup. Calm.

Response shape

Every response is JSON with a flat shape — { data, error } on writes, a plain array on reads. No envelopes, no HATEOAS. Errors return HTTP status + { error: "human-readable message" }.

Type scale

  • Display — Fraunces, weight 300, italic for emphasis, opsz 144 at large sizes (more contrast in the curves)
  • Body — IBM Plex Sans, weight 300 (paragraphs) / 400 (UI labels) / 500 (strong)
  • Code & meta — IBM Plex Mono, weight 400, 0.04–0.22em letter-spacing for caps
  • Sizes — 11, 12.5, 13.5, 15.5 (body), 18, 22, 32, 44, 64 — a quarter-step scale, not a half-step

Motion system

Animations exist to confirm, not to entertain. Three timing curves cover everything:

  • Reveal — cubic-bezier(.2, .7, .25, 1), 500 ms, used for page/section entry with 40 ms stagger between siblings
  • Hover — ease-out, 250–350 ms, lifts cards by 3 px and adds a 30 px soft shadow
  • Confirm — ease-out, 1600 ms total, used by the Copy button: label fades to checkmark in 200 ms, sits in "copied" state, fades back

No parallax. No spring physics. No micro-interactions on every element. The motion budget is intentionally small.

09 — Build & ship

One command produces six binaries.

Because we picked pure-Go SQLite, there is no CGO and cross-compilation is trivial. A single make release produces every binary for macOS, Linux, and Windows on both amd64 and arm64.

Makefilemake
VERSION := $(shell git describe --tags --always)
LDFLAGS := -s -w -X main.version=$(VERSION)
TARGETS := \
  darwin/amd64 darwin/arm64 \
  linux/amd64  linux/arm64  \
  windows/amd64 windows/arm64

dev:
	go run ./...

release:
	@for t in $(TARGETS); do \
	  GOOS=$${t%/*} GOARCH=$${t#*/} \
	  go build -trimpath -ldflags "$(LDFLAGS)" \
	    -o dist/prompt-vault-$${t%/*}-$${t#*/}$$( \
	      [ $${t%/*} = windows ] && echo .exe ) .; \
	done

clean:
	rm -rf dist/

Each binary is ~12 MB stripped. The first run creates ~/.prompt-vault/vault.db, opens a browser tab at http://127.0.0.1:7777, and starts the fsnotify watchers. Ctrl-C flushes the WAL and exits clean.

11 — Trade-offs

What we're not building.

Every decision implies a refusal. These are the ones to name now so we don't drift later:

  • No sync. One machine, one vault. Adding sync turns a tool into a service. If users ask, suggest committing vault.db to a private git repo.
  • No accounts, no telemetry. The binary never makes an outbound request. Ever.
  • No prompt running. This is a library, not an IDE. We don't call OpenAI/Anthropic. Copy is the only output.
  • No teams. If you need shared prompts, this is the wrong tool. Reach for Langfuse or PromptHub.
  • No mobile. The Atelier design is desktop-first and won't compromise to fit a phone. A read-only mobile view is M5+.
  • No embedding-based search until FTS5 demonstrably falls short. The bar for adding that complexity is high.
The product, in one line

A local, calm, single-binary library that turns the prompts scattered across your sessions and bookmarks into a place you'll actually return to.
