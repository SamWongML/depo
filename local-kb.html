§ 01 — Brief

The build brief, sharpened.

You're not picking a product. You're building a service. It runs locally on a MacBook Pro behind a corporate proxy, it costs nothing to run, and it does three jobs:

  • Knowledge base. Ingest official docs, code snippets, audits, prompts, ADRs, lessons-learned. Hybrid-searchable, payload-filterable, versioned.
  • Agent memory. Capture conversations with coding agents — what they tried, what worked, what failed. Time-weighted. Importance-weighted. Deduplicated.
  • Two-faced API. A typed REST + SSE surface for the production-grade HTML frontend you will build, and an MCP server surface for the coding agents to read and write through.

The frontend is out of scope for this report — that's your work. But the API is designed so the frontend story is trivial: FastAPI emits OpenAPI 3.1, which generates fully-typed TypeScript clients via openapi-typescript in one command. You get IDE autocomplete on every endpoint, every field, every error response.

§ 02 — Language

Why Python, and why decisively.

Four languages had to be considered: Python, TypeScript/Node, Go, Rust. The honest answer is that Python wins, and not by a small margin. Here's the comparison stripped to what matters for this exact workload:

Capability Python TypeScript Go Rust
Embedding models (BGE-M3, Qwen3, Nomic) native, all of them transformers.js, slower FFI only FFI only
Cross-encoder rerankers native limited FFI only candle, growing

Raw HTTP throughput ~5k rps ~10k rps ~40k rps ~50k rps
Inference dominates request time Yes — by 100–500×. Framework overhead is rounding error.

The three Python advantages that close it.

1 · The ML ecosystem is Python-native.

Every embedding and reranker you'd actually use ships first as PyTorch + a HuggingFace config. FlagEmbedding (the official BGE-M3 library) only exists in Python. sentence-transformers is Python. FastEmbed has bindings but is Python-led. Document parsers — Docling, unstructured, LlamaIndex readers — are Python-first by months. Choosing TypeScript here means writing wrappers around Python subprocesses. Don't.

2 · FastAPI + Pydantic v2 is best-in-class for a typed API.

FastAPI emits OpenAPI 3.1 from your Pydantic models with zero extra work. Your frontend runs npx openapi-typescript http://localhost:8000/openapi.json -o api.d.ts and gets full type definitions for every request, response, and error. Pydantic v2 (Rust core) does field validation in microseconds. Async I/O is mature. Streaming responses (SSE) are a one-liner. This isn't a Python apology; it's the right tool.

3 · FastMCP collapses the agent surface to decorators.

The official Python MCP SDK has been wrapped by FastMCP, which gives you a FastAPI-style decorator API (@mcp.tool()) that infers the JSON schema from your type hints. Most of the dependents on PyPI for mcp pull it in transitively through FastMCP. A complete MCP server for this knowledge base is roughly 80 lines.

The Node case, fairly stated

If your frontend is Next.js and you want one language end-to-end, the AI-first pattern in 2026 is Python service + Node gateway: keep the KB in Python, put a thin Node gateway in front that adds auth, streaming UX, and shares types with the frontend. It works. It's just two services to maintain instead of one.

§ 03 — Architecture

Reference architecture, top to bottom.

One Python process, two sidecars (Qdrant in Docker, optionally Ollama if you want a local chat LLM in the loop). Each layer has one job. The seams are explicit so any layer can be swapped without rewriting the others — including the reranker and the embedding model.

What lives where, and why.

Qdrant — the heavy lifting

All the vectors and their payloads. Hybrid queries run server-side; you ship one request and get a fused, filtered, ranked list back. Snapshots are a one-call API so backups are trivial. Bind it to 127.0.0.1.

SQLite — the source of truth

Document-level metadata (source, hash, version, owner), the audit log (every write with who/when/why), the ingest queue, eval-run history. SQLite in WAL mode handles thousands of writes/sec and the whole DB is one file to back up.

The dual-store rule: Qdrant is the index, SQLite is the system of record. If Qdrant gets corrupted or you change embedding models, you reconstruct it from SQLite by re-embedding. Never make Qdrant the only place a fact lives.

§ 04 — Storage

Storage layer: Qdrant collection schema & SQLite tables.

The Qdrant collection

Three named vectors in one collection, with a rich payload schema. This is the configuration that enables true hybrid search in a single query: 

from qdrant_client import QdrantClient, models

client = QdrantClient(url="http://localhost:6333")

client.create_collection(
    collection_name="kb",
    vectors_config={
        "dense": models.VectorParams(size=1024, distance=models.Distance.COSINE),
        "colbert": models.VectorParams(
            size=128,
            distance=models.Distance.COSINE,
            multivector_config=models.MultiVectorConfig(
                comparator=models.MultiVectorComparator.MAX_SIM,
            ),
        ),
    },
    sparse_vectors_config={
        "sparse": models.SparseVectorParams(modifier=models.Modifier.IDF),
    },
    # on-disk for collections > ~200k chunks; in-memory below
    optimizers_config=models.OptimizersConfigDiff(memmap_threshold=200_000),
    quantization_config=models.ScalarQuantization(
        scalar=models.ScalarQuantizationConfig(type=models.ScalarType.INT8, always_ram=True),
    ),
)

# Payload indices for fast filtering
for field, schema in [
    ("project",  models.PayloadSchemaType.KEYWORD),
    ("type",     models.PayloadSchemaType.KEYWORD),
    ("tags",     models.PayloadSchemaType.KEYWORD),
    ("created_at", models.PayloadSchemaType.DATETIME),
    ("importance", models.PayloadSchemaType.FLOAT),
    ("doc_id",   models.PayloadSchemaType.KEYWORD),
]:
    client.create_payload_index("kb", field_name=field, field_schema=schema)

Why three named vectors

  • dense (1024-dim) — BGE-M3 dense output; semantic recall.
  • sparse (IDF-modified) — BGE-M3 sparse / lexical output; literal-token recall.
  • colbert (multi-vector, 128-dim per token) — BGE-M3 ColBERT output; late-interaction precision.

Scalar INT8 quantization is non-optional at scale: it cuts memory ~4× with <1% recall loss. memmap_threshold pushes the index to disk past 200K chunks so you don't eat RAM you need for the embedder.

The SQLite tables

Keep this small. Six tables cover everything. Use Pydantic models as the canonical definition and SQLModel (or plain SQLAlchemy) to materialize them.

Idempotency, baked in

documents.content_sha + chunks_meta.embed_ver let you skip re-embedding unchanged content on every ingest run. Compute SHA-256 of the source body before chunking; if it matches a non-deleted document, return its id and exit.

§ 05 — Embeddings

Embedding layer: BGE-M3, all three outputs at once.

The architectural punchline of BGE-M3 is that a single forward pass returns dense, sparse (lexical), and ColBERT multi-vector representations. That's what makes the three-named-vector Qdrant collection work without three separate model invocations. 

# embed/bge_m3.py
from FlagEmbedding import BGEM3FlagModel
from qdrant_client import models
from functools import lru_cache

@lru_cache(maxsize=1)
def get_model() -> BGEM3FlagModel:
    # use_fp16 cuts VRAM ~2× with negligible quality loss; auto-uses MPS on Apple Silicon
    return BGEM3FlagModel("BAAI/bge-m3", use_fp16=True)

def embed_chunks(texts: list[str], batch_size: int = 12) -> list[dict]:
    out = get_model().encode(
        texts,
        batch_size=batch_size,
        max_length=8192,            # BGE-M3 supports 8K tokens natively
        return_dense=True,
        return_sparse=True,
        return_colbert_vecs=True,
    )
    return [
        {
            "dense": out["dense_vecs"][i].tolist(),
            "sparse": models.SparseVector(
                indices=list(out["lexical_weights"][i].keys()),
                values=list(out["lexical_weights"][i].values()),
            ),
            "colbert": out["colbert_vecs"][i].tolist(),
        }
        for i in range(len(texts))
    ]

Why not Qwen3-Embedding or Nomic v2 here?

Qwen3-Embedding-8B scores higher on raw MTEB (70.58 multilingual, 80.68 on MTEB Code). It's a candidate for a code-search-specialist variant of the system. But it only emits dense vectors. To do hybrid with it you'd run BM25 on the side via rank_bm25 or build your own SPLADE pipeline — that's two more moving parts, two more models in RAM, and a custom fusion implementation. BGE-M3 collapses that to one forward pass per chunk.

Performance budget on a MacBook Pro M-series

  • BGE-M3 (fp16, MPS): ~40–60 chunks/sec at batch 12, 500-token chunks.
  • RAM cost: ~2.5 GB resident.
  • 100K-chunk corpus indexes in ~30–40 minutes — one coffee.
§ 06 — Retrieval

The retrieval pipeline, in one server-side call.

The 2026 SOTA pattern, validated in the corrective-RAG benchmark paper and across Qdrant / Elastic / Zep production deployments, is three stages: multi-recall → RRF fusion → cross-encoder rerank. Qdrant fuses for you; you only run the reranker. 

            QUERY ("how do we handle 429s in the SDK")
              │
              ▼
   embed once with BGE-M3 → {dense, sparse, colbert}
              │
              ▼
   ┌──────────────────────────────────────────────────────────────┐
   │ Qdrant query_points  (one HTTP call)                         │
   │  prefetch: dense(top 100), sparse(top 100), colbert(top 50)  │
   │  fusion:   RRF (k=60)                                        │
   │  filter:   project="my-api" AND type IN (doc,best-practice)  │
   │  limit:    25 fused candidates                               │
   └──────────────────────────────────────────────────────────────┘
              │
              ▼
   cross-encoder rerank — bge-reranker-v2-m3 over (query, doc) pairs
              │
              ▼
   apply memory-dynamics boost — importance + recency (§ 07)
              │
              ▼
   TOP 5 → coding agent / frontend

# retrieval/hybrid.py
from qdrant_client import models
from FlagEmbedding import FlagReranker

reranker = FlagReranker("BAAI/bge-reranker-v2-m3", use_fp16=True)

def hybrid_search(
    query: str,
    *,
    project: str | None = None,
    types: list[str] | None = None,
    top_k: int = 5,
    candidates: int = 25,
) -> list[dict]:
    qv = embed_chunks([query])[0]
    flt = _build_filter(project=project, types=types)

    fused = client.query_points(
        collection_name="kb",
        prefetch=[
            models.Prefetch(query=qv["dense"],   using="dense",   limit=100, filter=flt),
            models.Prefetch(query=qv["sparse"],  using="sparse",  limit=100, filter=flt),
            models.Prefetch(query=qv["colbert"], using="colbert", limit=50,  filter=flt),
        ],
        query=models.FusionQuery(fusion=models.Fusion.RRF),
        query_filter=flt,
        limit=candidates,
        with_payload=True,
    ).points

    # Stage 3 — cross-encoder rerank
    pairs  = [(query, p.payload["text"]) for p in fused]
    scores = reranker.compute_score(pairs, normalize=True)

    # Stage 4 — memory-dynamics boost (importance × recency); see § 07
    blended = [(_blend(p, s), p) for p, s in zip(fused, scores)]
    blended.sort(key=lambda x: -x[0])

    return [_to_result(s, p) for s, p in blended[:top_k]]

Always return scores from every stage in the API response — dense_rank, sparse_rank, fused_score, rerank_score, final_score. Your frontend needs them for the "why did this rank here?" explainability view, and your eval harness needs them to diagnose regressions when you swap models.

Tunables that actually matter

RRF k: default 60, but published 2026 benchmarks find k=10 better on enterprise corpora — it weights top hits more. Reranker depth: 25–50 candidates is the sweet spot. Below 20, the reranker can't recover from poor recall; above 50, latency rises faster than quality.

§ 07 — Memory dynamics

What turns a KB into a memory.

A retrieval-augmented knowledge base treats every document as equal. A memory system treats them as weighted by importance, faded by time, and consolidated when redundant. The 2026 production patterns (Mem0, Letta/MemGPT, Zep/Graphiti) all encode some version of this. You should encode the minimal version.

Three dynamics, expressed as payload fields.

Dynamic 01Importance
A float in [0, 1] set on write. Best practices and audit findings get 0.9. Official doc excerpts get 0.7. Raw conversation chunks start at 0.3 and can be promoted by an explicit "this was useful" signal from the agent or user. Stored as a payload float, indexed, used as a search-time multiplier.
Dynamic 02Recency
A monotonic decay applied at query time, not at write. Cheap form: recency = exp(-Δdays / half_life) with half-life of 90 days for docs, 30 for conversations. Don't decay best-practices and ADRs — they're meant to be timeless. Compute server-side from created_at; no re-indexing ever needed.
Dynamic 03Consolidation
A background job that runs nightly: find clusters of near-duplicate chunks (cosine > 0.95 on dense), summarize each cluster into one canonical chunk, soft-delete the originals, link them via superseded_by in the audit log. Keeps the index lean and prevents the same fact from outvoting itself five times in retrieval.

The blend formula

def _blend(point, rerank_score: float) -> float:
    importance = point.payload.get("importance", 0.5)
    type_      = point.payload.get("type", "doc")
    age_days   = (utcnow() - parse(point.payload["created_at"])).days
    half_life  = 30 if type_ == "conversation" else 90
    recency    = 1.0 if type_ in {"best-practice", "adr"} else math.exp(-age_days / half_life)

    # Weighted blend — keep rerank dominant, let dynamics nudge ties
    return 0.70 * rerank_score + 0.20 * importance + 0.10 * recency
Should you add a graph?

Graphiti / Zep build a temporal knowledge graph for "what was true when" queries. It measurably wins on multi-hop benchmarks (Mem0g hits 68.4% on LOCOMO) but adds substantial complexity and requires LLM calls at ingest time. For personal coding-agent memory the flat hybrid + importance/recency model is enough. Add a graph only if you find yourself needing relational queries that vector + payload filters can't answer.

§ 08 — Ingestion

Ingestion pipeline.

Three classes of input dominate this use case: conversations (markdown export from Claude Code / Cursor), official docs (PDFs, HTML, markdown), and code-adjacent artifacts (ADRs, README, audit reports, prompts). Different parsers, different chunkers.

PDF / DOCX / HTML

Docling

IBM, MIT-licensed

Layout-aware. Preserves tables and code blocks. Outputs clean Markdown. The current best-in-class open parser for messy docs.

Markdown / text

Heading-aware splitter

~100 LOC, no deps

Respect H1/H2/H3 boundaries. Don't break in the middle of a code fence. 500–800 tokens per chunk, 80-token overlap.

Source code

tree-sitter chunker

function/class-level

Split at AST boundaries. Keep imports + signature with each chunk. Re-attach docstring to body. Massively improves code retrieval quality.

Conversations

Turn-pair chunker

custom

One chunk = one user message + the agent's response, plus the prior 1–2 turns as context. Tag with conversation id so you can reconstruct the session.

The ingest loop, idempotent

def ingest(source: Source, body: str, meta: dict) -> IngestResult:
    sha = sha256(body.encode()).hexdigest()
    # Skip if unchanged
    existing = db.find_document_by_sha(sha)
    if existing and not existing.deleted_at:
        return IngestResult(status="skipped", document_id=existing.id)

    doc = db.upsert_document(source=source, sha=sha, body=body, meta=meta)
    chunks = chunk_for(source.type, body)
    embeddings = embed_chunks([c.text for c in chunks])

    points = [
        models.PointStruct(
            id=str(uuid7()),
            vector={"dense": e["dense"], "colbert": e["colbert"]},
            sparse_vectors={"sparse": e["sparse"]},
            payload={
                "text": c.text, "doc_id": doc.id,
                "project": meta.get("project"), "type": source.type,
                "importance": default_importance(source.type),
                "created_at": utcnow().isoformat(),
                "tags": meta.get("tags", []),
            },
        )
        for c, e in zip(chunks, embeddings)
    ]
    qdrant.upsert("kb", points=points)
    db.upsert_chunks_meta(doc.id, points)
    db.audit("ingest", target=doc.id, payload={"chunks": len(points)})
    return IngestResult(status="ingested", document_id=doc.id, chunks=len(points))

Two Pydantic models, the whole API in your head

# api/models.py — what the frontend will see, typed end-to-end
from pydantic import BaseModel, Field
from datetime import datetime
from typing import Literal

DocType = Literal["audit", "conversation", "doc", "prompt", "adr", "lesson", "best-practice"]

class SearchRequest(BaseModel):
    query: str
    project: str | None = None
    types: list[DocType] | None = None
    tags: list[str] | None = None
    top_k: int = Field(5, ge=1, le=50)
    rerank: bool = True
    explain: bool = False

class SearchHit(BaseModel):
    point_id: str
    document_id: str
    text: str
    type: DocType
    project: str | None
    tags: list[str]
    created_at: datetime
    importance: float
    dense_rank: int | None = None
    sparse_rank: int | None = None
    fused_score: float
    rerank_score: float | None = None
    final_score: float

class SearchResponse(BaseModel):
    hits: list[SearchHit]
    took_ms: int
    query_id: str          # for telemetry / "was this helpful?" callback

# api/routes/search.py
from fastapi import APIRouter

router = APIRouter()

@router.post("/search", response_model=SearchResponse)
async def search(req: SearchRequest) -> SearchResponse:
    with timed() as t:
        hits = retrieval.hybrid_search(
            req.query, project=req.project, types=req.types,
            tags=req.tags, top_k=req.top_k, rerank=req.rerank,
        )
    query_id = audit.log_search(req, hits)
    return SearchResponse(hits=hits, took_ms=t.ms, query_id=query_id)

§ 10 — MCP

MCP server for coding agents.

Coding agents talk to the same service through MCP. Run it as a separate entry point that imports the same retrieval and ingestion modules — zero code duplication. FastMCP 2 turns this into decorators, and tool descriptions become the agent's instructions. 

# mcp/server.py — entry: `uv run kb-mcp` (registered in your pyproject)
from mcp.server.fastmcp import FastMCP
from kb.retrieval import hybrid_search
from kb.ingestion import ingest
from kb.api.models import SearchHit

mcp = FastMCP("kb")

@mcp.tool()
def find(query: str, project: str | None = None, top_k: int = 5) -> list[dict]:
    """Search the personal knowledge base for prior conversations, audits,
    best practices, ADRs, and official-doc excerpts.

    Use this BEFORE writing new code that touches existing functionality.
    Use this BEFORE answering questions about project conventions.
    Always pass `project` if you know which project you're working in.
    """
    hits = hybrid_search(query, project=project, top_k=top_k)
    return [h.model_dump() for h in hits]

@mcp.tool()
def remember(
    information: str,
    type: str = "lesson",
    project: str | None = None,
    tags: list[str] = [],
    importance: float = 0.5,
) -> dict:
    """Store a lesson, best practice, prompt, or audit finding for future retrieval.
    Use this when you discover something worth remembering across sessions:
    a non-obvious bug fix, a project convention, a working prompt pattern.
    Set importance > 0.7 only for facts you want surfaced often.
    """
    res = ingest(
        Source(type=type, uri=f"mcp://agent/{utcnow().isoformat()}"),
        body=information,
        meta={"project": project, "tags": tags, "importance": importance},
    )
    return {"document_id": res.document_id, "status": res.status}

if __name__ == "__main__":
    mcp.run()  # stdio transport for Claude Code / Cursor / Cline

Why tool docstrings matter

The agent reads the docstring to decide when to call your tool. Write it as instructions, not documentation. "Use this BEFORE…" beats "This function returns…". When you see your agent forget to use the memory, the fix is almost always in the docstring, not the code.

§ 11 — Observability

Observability, monitoring & eval.

Production-grade means you can answer three questions at any time: what is the service doing right now?, what did it do last week?, and did my last change make retrieval better or worse?

1 · Live metrics — Prometheus exposition

Use prometheus-client. Expose at /metrics. The frontend dashboard can read it directly or via a tiny /stats/snapshot aggregation endpoint.

  • kb_search_seconds — histogram, labels: rerank=true|false, project.
  • kb_embed_seconds — histogram, labels: kind=query|doc.
  • kb_ingest_chunks_total — counter, labels: source_type.
  • kb_qdrant_collection_points — gauge.
  • kb_errors_total — counter, labels: route, kind.

2 · Audit log — append-only, queryable

Every write hits audit_log with actor (frontend / mcp:client-name / ingest-job-id), action, target, and the full payload. Powers your frontend's "activity feed" and "who changed this?" view. SQLite handles 10K writes/sec; you'll never feel it.

3 · Eval harness — regression-test search quality

The single biggest risk in a memory system is silent quality regression. You swap embedding models, change RRF k, tweak the blend formula — and search gets quietly worse. A small golden-set eval catches it.

  • Maintain ~/.kb/evals/golden.jsonl — 30–100 (query, expected_doc_id) pairs you curate over time.
  • POST /eval/runs runs the full set, records recall@5 and nDCG@5 per query, stores the result in eval_runs.
  • Your frontend renders this as a sparkline with current vs previous run; any change >3pp warrants investigation.
Structured logs, please

Use structlog with JSON output. Every log line carries query_id (from the search response) or ingest_job_id. When a user reports "this search felt off", they paste the query_id and you can reconstruct every decision the pipeline made.

§ 13 — Corp network

Corporate-network checklist.

  1. Set HTTPS_PROXY in your shell and inside Docker. Never set HTTP_PROXY — it breaks Ollama's localhost client.
  2. Export one CA bundle path and reuse it everywhere: SSL_CERT_FILE (Python/uv/requests), REQUESTS_CA_BUNDLE, CURL_CA_BUNDLE, NODE_EXTRA_CA_CERTS (for your frontend tooling).
  3. Pre-warm the HuggingFace cache from a machine with reach: HF_HUB_OFFLINE=1 after copying ~/.cache/huggingface over.
  4. Bind Qdrant to 127.0.0.1:6333, never 0.0.0.0 — keeps it off the office network.
  5. Mirror qdrant/qdrant:latest to your internal Docker registry if Docker Hub is blocked; pin the SHA in docker-compose.yml.
  6. Use uv pip install --index-url with your internal PyPI mirror; check it in via uv.lock.
  7. For your frontend dev server, add the FastAPI origin to CORS allow-list. In dev, http://localhost:5173; in "production" on the same Mac, http://localhost:8000 serves both.
§ 14 — Build sequence

The build sequence, two-week version.

Day 1–2Foundations
uv init kb. Docker-compose up Qdrant. Bootstrap the collection with three named vectors and payload indices. Get BGE-M3 loading and emitting all three vector types. Get one chunk written and read back. SQLite migrations in place.
Day 3–4Retrieval
Implement the hybrid search function. Add the reranker. Add the memory-dynamics blend. Smoke-test against 100 hand-written chunks. Pin recall@5 ≥ 0.9 on your golden set before moving on.
Day 5–7API + ingestion
FastAPI app, routes, Pydantic models. Docling + markdown + tree-sitter chunkers. The idempotent ingest pipeline. SSE for ingest progress. Confirm OpenAPI 3.1 spec generates and TypeScript types come out clean.
Day 8–9MCP + observability
FastMCP server with find and remember tools. Prometheus metrics. Audit log writer. Wire into Claude Code; verify the agent calls find spontaneously when context is missing.
Day 10Eval harness
Build the golden-set runner. Persist results to eval_runs. Now you can tune blend weights and reranker depth with data, not intuition.
Day 11–12Consolidator + backups
Nightly dedup job. Qdrant snapshot script. SQLite hourly copy to ~/.kb/backups/. Now you can lose your laptop and recover.
Day 13–14Hardening
API token middleware. Per-route rate limiting. Structured logging with query_id propagation. Read-only mode for ingest windows. Now hand it off to your frontend.
