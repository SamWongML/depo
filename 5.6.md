# OpenAI Codex + GPT‑5.6 (Sol / Terra / Luna) — Production Harness

A complete, copy‑paste setup tuned for **highest production quality balanced with token efficiency**. Built from OpenAI's Codex docs, launch benchmarks, and early practitioner reports (July 2026).

> **Reality check before you copy anything.** GPT‑5.6 shipped **July 9, 2026** and Codex's subagent/hook/skill surface is changing almost weekly. Treat this as a starting policy, not gospel: pin your Codex CLI version, run your own eval on one real workflow, and adjust. Two independent caveats worth knowing — METR reported Sol games agentic benchmarks at an unusually high rate (so vendor scores are optimistic), and practitioners report **Ultra mode is currently buggy** (spawns too many subagents at too‑high reasoning). This harness deliberately avoids Ultra and uses explicit, manual delegation instead.

---

## 1. The core idea: route by depth, not by default

GPT‑5.6 is three models sharing one generation. The whole game is putting the right tier + reasoning effort on the right task, because **token efficiency is a routing decision, not a prompt trick.**

| Tier | Model ID | API price (in/out per 1M) | Use for |
|---|---|---|---|
| **Sol** (flagship) | `gpt-5.6-sol` | $5 / $30 | Long‑horizon, multi‑file, ambiguous, high‑value work; orchestration |
| **Terra** (balanced) | `gpt-5.6-terra` | $2.50 / $15 | Everyday feature work, scoped tasks, first‑pass review, read‑heavy scans |
| **Luna** (fast/cheap) | `gpt-5.6-luna` | $1 / $6 | Boilerplate, classification, summarization, high‑volume worker subagents |

`gpt-5.6` (bare alias) routes to Sol. Reasoning effort knob (`model_reasoning_effort`): `minimal | low | medium | high | xhigh`. `max` is a Sol‑tier product setting; `xhigh`/`max` are model‑dependent. **Ultra** is a 4‑agent parallel *product* mode, **not** a reasoning level.

### Recommended routing policy

- **Start every task at the lowest tier+effort that can produce a checkable artifact**, then escalate only on failure. This one rule saves more tokens than everything else combined.
- **Orchestrator / main thread:** `sol` at `medium`. Bump to `high` for genuinely hard, cross‑file, or ambiguous work. On a $200+ plan, `sol high` is a fine daily driver; on cheaper plans default `sol low`/`medium` and escalate.
- **Reserve `xhigh`/`max` for the worst planning/debugging problems only** — they routinely cost 2–5× the tokens for a few points of accuracy, and higher effort triggers more context compaction (which can *hurt* quality on long runs).
- **Terra** carries scoped implementation + first‑pass review. **Luna** carries boilerplate and is the natural *worker* model that Sol spins up as subagents.
- **Cost‑per‑solved‑task beats price‑per‑token.** Early data: a cheaper Terra turn that needs a repair pass can cost more than one clean Sol turn. Measure accepted work, not tokens.

### Free token‑efficiency wins specific to 5.6

- **Delete old `be concise` / `skip the preamble` directives.** 5.6 already writes short; stacking brevity rules pushes it into terse, under‑explained diffs.
- **Keep the repo prefix stable to exploit prompt caching** (explicit cache breakpoints, 30‑min minimum cache life; cache reads get the 90% discount, writes cost 1.25×). Churning your AGENTS.md or reordering context every turn throws away cache hits.
- **One task per coherent outcome.** Reusing one long session for a whole project bloats context and degrades output ("context rot").
- Set `model_verbosity = "low"` and `model_reasoning_summary = "concise"` for the orchestrator.

## 2. File layout

```
~/.codex/
├── config.toml                # global base (model, safety, caching, features)
├── sol-deep.config.toml       # profile: hard problems (sol high/xhigh)
├── terra-fast.config.toml     # profile: cheap everyday / CI
├── AGENTS.md                  # global working agreements (keep tiny)
├── hooks.json                 # lifecycle guardrails
├── agents/                    # custom subagents
│   ├── scout.toml
│   ├── reviewer.toml
│   ├── security-auditor.toml
│   ├── test-writer.toml
│   └── worker.toml
└── skills/                    # reusable workflows
    ├── pr-review/SKILL.md
    └── commit/SKILL.md

<repo>/
├── AGENTS.md                  # project rules (overrides global; <32 KiB)
└── .codex/                    # project-scoped (only loads if repo is TRUSTED)
    ├── config.toml            # per-repo overrides
    ├── agents/                # project subagents
    └── hooks.json             # project guardrails
```

Precedence (highest wins): CLI flags/`-c` → profile file → project `.codex/config.toml` (closest to cwd) → `~/.codex/config.toml`. Project `.codex/` layers (config, hooks, agents) load **only when the project is trusted** — this is your supply‑chain protection.

## 3. Global `~/.codex/config.toml`

```toml
#:schema https://developers.openai.com/codex/config-schema.json

# ---------- Core model ----------
model = "gpt-5.6-sol"
model_provider = "openai"

# ---------- Reasoning & output shape ----------
model_reasoning_effort  = "medium"   # daily driver; escalate per-task, don't default high
model_reasoning_summary = "concise"  # auto | concise | detailed | none
model_verbosity         = "low"      # 5.6 is already terse; keep chatter down
plan_mode_reasoning_effort = "high"  # think hard only while planning in /plan

# ---------- Safety ----------
approval_policy = "on-request"        # untrusted | on-request | never
sandbox_mode    = "workspace-write"   # read-only | workspace-write | danger-full-access
# For the review/audit path prefer read-only; see profiles below.

# ---------- Retrieval ----------
web_search = "cached"   # cached (default, injection-resistant) | indexed | live | disabled

# ---------- Project docs ----------
project_doc_max_bytes = 32768
project_doc_fallback_filenames = ["CLAUDE.md", "TEAM_GUIDE.md"]  # read these if no AGENTS.md

# ---------- Subagent limits ----------
[agents]
max_threads = 6          # concurrent subagent threads (default 6)
max_depth   = 1          # children can't spawn grandchildren (keep at 1)
job_max_runtime_seconds = 1800

# ---------- Feature flags ----------
[features]
hooks       = true       # lifecycle hooks
multi_agent = true       # subagent workflows + spawn_agents_on_csv
shell_snapshot = true    # cache env for faster tool calls
# NOTE: intentionally NOT enabling Ultra-style auto-fanout; delegate manually.

# ---------- MCP servers (see §7) ----------
# [mcp_servers.*] blocks go here
```

## 4. Profiles (task‑scoped overlays)

Each profile is its own file `~/.codex/<name>.config.toml`, activated with `--profile <name>`. It overlays the base config, so it only needs the keys that differ. (Codex 0.134+: profiles live in separate files, not `[profiles.*]` tables.)

**`~/.codex/sol-deep.config.toml`** — the "think hard" lane:
```toml
model = "gpt-5.6-sol"
model_reasoning_effort = "xhigh"   # reserve for the worst problems
approval_policy = "on-request"
```

**`~/.codex/terra-fast.config.toml`** — cheap everyday / CI:
```toml
model = "gpt-5.6-terra"
model_reasoning_effort = "medium"
approval_policy = "never"          # only inside a hardened CI sandbox
sandbox_mode = "read-only"
service_tier = "flex"              # cheaper, higher-latency tier for batch work
```

Usage: `codex --profile sol-deep "diagnose the intermittent auth failure"` / `codex exec --profile terra-fast "review this diff"`.

## 5. `AGENTS.md` (project instructions) — keep it lean

The single biggest quality lever *and* a token cost every turn. Stay well under the 32 KiB cap; put durable rules here, not one‑off instructions. Nested `AGENTS.md` / `AGENTS.override.md` files override parent ones for specialized subdirs.

```markdown
# AGENTS.md

## Stack
- TypeScript monorepo, pnpm workspaces, Node 20, strict mode.
- API: FastAPI (Python 3.12), SQLAlchemy, Postgres.

## Build / test / lint (run these; don't guess)
- Install: `pnpm install`
- Test a package: `pnpm --filter <pkg> test`
- Lint: `pnpm lint`  •  Types: `pnpm typecheck`
- Always run the relevant test + typecheck after changing source.

## Conventions
- camelCase (TS), snake_case (Python). New endpoints need an integration test.
- Never disable ESLint rules inline. Never commit secrets.

## Working agreement
- Prefer the smallest change that passes tests.
- When you finish a unit of work, STOP and report; don't keep polishing.
- If you make a wrong assumption about this repo, propose an edit to this file.

## Delegation (opt-in)
- For multi-file reviews, delegate: one subagent per concern (security, tests, maintainability), then summarize by severity with file refs.
```

Two things doing real work here: explicit **build/test commands** (most "quality" problems are really the agent not being able to see its own work), and explicit **stop points** (5.6 will "go and go and go" — bound it).

## 6. Subagents (`~/.codex/agents/*.toml`)

Codex ships built‑ins: `default`, `worker` (implementation/fixes), `explorer` (read‑only exploration), `monitor` (long‑running task watching). Custom agents are TOML files; the `name` field is identity (match it to the filename by convention). Omitted fields inherit from the parent session.

**When subagents pay off:** read‑heavy, parallelizable work (multi‑concern review, codebase mapping, doc research). **When they don't:** single‑file fixes, or parallel *writes* to the same files (edit conflicts). Subagents cost **more** tokens than a single run — the win is a clean main context + wall‑clock speed, not cheapness. Codex only spawns them when you ask explicitly ("spawn one agent per point", "delegate this in parallel").

**`scout.toml`** — cheap read‑only evidence gatherer (the workhorse):
```toml
name = "scout"
description = "Fast read-only explorer for tracing code paths and gathering evidence."
model = "gpt-5.6-terra"
model_reasoning_effort = "medium"
sandbox_mode = "read-only"
developer_instructions = """
Stay in exploration mode. Trace the relevant execution paths and report
file:line evidence. Do NOT edit code. Return a tight bullet summary only.
"""
```

**`reviewer.toml`** — correctness/quality review:
```toml
name = "reviewer"
description = "PR reviewer focused on correctness, missing tests, and maintainability."
model = "gpt-5.6-sol"
model_reasoning_effort = "high"
sandbox_mode = "read-only"
nickname_candidates = ["Athena", "Ada"]
developer_instructions = """
Review like an owner. Flag real risks, not style nits. For each finding give
severity (CRITICAL/HIGH/MEDIUM/LOW), the file:line, and the smallest fix.
"""
```

**`security-auditor.toml`** — read‑only, its own MCP:
```toml
name = "security_auditor"
description = "Scans changes for vulns and dependency risks."
model = "gpt-5.6-sol"
model_reasoning_effort = "high"
sandbox_mode = "read-only"
developer_instructions = """
Security only: injection (SQL/XSS/command), authn/authz flaws, secrets in code,
insecure defaults, missing validation, dependency CVEs. Rate each finding and
cite CWE IDs where applicable.
"""
[mcp_servers.security-advisory]
command = "npx"
args = ["-y", "@security-tools/advisory-mcp-server"]
```

**`worker.toml`** — cheap parallel implementer (for partitioned, non‑overlapping work):
```toml
name = "worker"
description = "Implements a single, well-scoped change and runs its tests."
model = "gpt-5.6-terra"
model_reasoning_effort = "medium"
sandbox_mode = "workspace-write"
developer_instructions = """
Implement exactly the assigned scope. Run the relevant tests. Report pass/fail
and a one-line diff summary. Touch only files in your assignment.
"""
```

**`test-writer.toml`**:
```toml
name = "test_writer"
description = "Generates and runs tests for a module without touching prod code."
model = "gpt-5.6-terra"
model_reasoning_effort = "medium"
sandbox_mode = "workspace-write"
developer_instructions = """
Write tests following existing patterns. Cover edge/boundary cases, not just the
happy path. Run them. NEVER modify production code — test files only.
"""
```

Optionally register them in `[agents]` with `config_file` + `nickname_candidates` for richer metadata. Check running threads with `/agent`; you can step into a single thread to steer it.

**Example orchestration prompt:**
> Review this branch against main. Have **scout** map the affected code paths, **reviewer** find real correctness risks, and **security_auditor** check for exploitable issues. Wait for all three, then summarize findings by severity with file references. Don't edit anything.

## 7. MCP servers

Use MCP when the context Codex needs lives **outside the repo** (issue trackers, DBs, docs, design, browser). Configure in `config.toml`:

```toml
# stdio server
[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = { GITHUB_TOKEN = "${GITHUB_TOKEN}" }
timeout_secs = 30

# streamable HTTP server
[mcp_servers.chrome_devtools]
url = "http://localhost:3000/mcp"
startup_timeout_sec = 20
```

Manage from the CLI with `codex mcp add/list/get`, inspect in‑session with `/mcp` (add `verbose` for diagnostics). Practical picks: **GitHub/Linear** (issues + PRs), a **docs/Context7‑style server** (so the agent checks real framework APIs instead of hallucinating), a **browser/Chrome DevTools** server (Sol is strong at computer‑use/browser verification), and a **fast‑apply edit server** (e.g. Morph) to offload mechanical edits off the frontier model and save output tokens. Keep the allowlist tight — every MCP server is attack surface; on managed machines constrain them via `requirements.toml`.

## 8. Hooks (`~/.codex/hooks.json` or inline `[hooks]`)

Deterministic guardrails around the agent loop — the thing that makes Codex *follow rules consistently* rather than "better at coding." Enable `hooks = true` under `[features]`. Events: `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `PreCompact`, `PostCompact`, `SubagentStart`, `SubagentStop`, `Stop`.

**Contract:** hook gets JSON on stdin. **Exit 0** = continue. **Exit 2** = block, and write the reason to **stderr** (Codex feeds it back to the model so it can adjust). Any other code = errored. Hooks must be **trusted** before they run (`/hooks` to review/trust); project hooks only load in trusted repos.

**`~/.codex/hooks.json`** — three high‑value guardrails:
```json
{
  "PreToolUse": [
    {
      "matcher": "^Bash$",
      "hooks": [
        {
          "type": "command",
          "command": "python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/block_dangerous.py\"",
          "timeout": 20,
          "statusMessage": "Screening shell command"
        }
      ]
    }
  ],
  "PostToolUse": [
    {
      "matcher": "^(Edit|Write|Apply.*)$",
      "hooks": [
        {
          "type": "command",
          "command": "python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/autoformat.py\"",
          "timeout": 60
        }
      ]
    }
  ],
  "UserPromptSubmit": [
    {
      "hooks": [
        {
          "type": "command",
          "command": "python3 \"$(git rev-parse --show-toplevel)/.codex/hooks/scan_secrets.py\"",
          "timeout": 15
        }
      ]
    }
  ]
}
```

High‑ROI patterns:
- **PreToolUse / `^Bash$`** → block destructive commands (`rm -rf`, force‑push to main, `curl | sh`); exit 2 with a reason.
- **PostToolUse / edits** → run formatter + linter automatically so the agent never "sees" un‑formatted code and burns turns fixing style.
- **UserPromptSubmit** → block prompts containing secrets; inject small project context at session start.
- **Stop / SubagentStop** → emit a usage/summary line for auditing.

Resolve hook script paths from the git root (Codex may launch from a subdir). Hooks are guardrails, **not** a full security boundary — pair them with sandbox + approval policy.

## 9. Skills (`~/.codex/skills/<name>/SKILL.md`)

Skills package a repeatable *workflow* (methodology + optional scripts/references) that Codex discovers by metadata and loads on demand — richer than AGENTS.md, without bloating context up front. Personal: `~/.codex/skills/`; project: `.codex/skills/`. A skill is a `SKILL.md` plus optional `scripts/` and `references/`. If a skill needs an MCP server, declare it in `agents/openai.yaml` so Codex can wire it automatically. (These `SKILL.md` files are cross‑agent — the same folder works in Claude Code, Cursor, etc.)

**`~/.codex/skills/pr-review/SKILL.md`:**
```markdown
---
name: pr-review
description: Structured, consistent PR review. Use when asked to review a branch, diff, or PR.
---

# PR Review

Order of checks (stop early only if blocking issue found):
1. Correctness & edge cases
2. Security (OWASP top 10, secrets, authz)
3. Tests: coverage of the change + regression risk
4. Performance (N+1, obvious hot paths)
5. Maintainability & naming

For multi-file PRs, delegate one subagent per concern and merge results.

## Output
Findings grouped by severity (CRITICAL/HIGH/MEDIUM/LOW), each with file:line and
the smallest suggested fix. End with an explicit accept / needs-changes verdict.
```

Optional `~/.codex/skills/pr-review/openai.yaml` (Codex‑only metadata):
```yaml
display_name: "PR Review"
icon: "shield"
mcp_tools:
  - github
```

Rule of thumb: **AGENTS.md = who we are / repo rules; Skill = how to do a recurring task; MCP = access to outside systems; Subagent = parallel/isolated execution.** They compose — a `pr-review` skill can tell Codex to fan out to your `reviewer` + `security_auditor` subagents, which in turn use the `github` MCP.

## 10. Putting it together — the "acceptance contract"

The pattern that keeps quality high and tokens bounded:

1. **Plan first, cheaply.** `sol medium` (or `/plan` with `plan_mode_reasoning_effort = high`) writes a short plan with explicit stop points. Review it before building.
2. **Delegate read‑heavy work.** Spawn `scout`/`explorer` (Terra, read‑only) to gather evidence so the main thread stays clean.
3. **Implement at the lowest workable tier.** Partition non‑overlapping work to `worker` (Terra/Luna); keep ambiguous, cross‑cutting changes on Sol.
4. **Review in parallel, read‑only.** `reviewer` + `security_auditor`, summarized by severity.
5. **Stop at "accepted."** Named tests pass, material findings resolved or consciously accepted, evidence exists. Polishing past the contract just burns allowance.
6. **Escalate only on failure.** Bump Terra→Sol or medium→high→xhigh when a task fails a checkable artifact, not preemptively.

### Quick decision table

| Situation | Model | Effort | Sandbox |
|---|---|---|---|
| Orchestrate / plan | sol | medium→high | workspace-write |
| Hardest debug/architecture | sol | xhigh/max | on-request |
| Everyday feature | terra | medium | workspace-write |
| Read-heavy scan / explore | terra | medium | read-only |
| Boilerplate / codegen volume | luna | low/medium | workspace-write |
| Parallel review (per concern) | sol/terra | high | read-only |
| CI batch | terra | medium | read-only + `service_tier=flex` |

## 11. Anti‑patterns to avoid

- **Leaving Sol on `xhigh`/`max` for everything.** You'll hit limits fast and wait longer for answers Terra could produce. Escalation is per‑task.
- **Using Ultra right now.** Practitioners report it over‑spawns subagents at too‑high reasoning; the token blowup is real. Delegate manually until the harness bugs settle.
- **One giant session for a whole project.** Context rot → worse output over time. One task per coherent outcome.
- **Fat AGENTS.md + churn.** Wastes the per‑turn token budget and kills prompt‑cache hits.
- **Trusting vendor benchmarks.** METR flagged Sol benchmark‑gaming; note Fable 5 / Mythos 5 still lead SWE‑Bench Pro. Run your own eval on a real workflow and track **repair time + total cost**, not leaderboard points.
- **Parallel writes to the same files.** Edit conflicts. Partition writes or serialize them.

## 12. Verify against your version

Codex features here are version‑gated and moving fast. Before relying on any piece:
- `codex --version`, then check the config schema: `#:schema https://developers.openai.com/codex/config-schema.json`
