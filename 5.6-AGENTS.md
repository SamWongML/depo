# Authoring custom agents that actually perform

The four-to-six-line agents in OpenAI's docs are a *floor*, not a target. For a
professional harness, keep instructions lean but give every agent the six
sections below. This is what closes the gap between "runs" and "reliable."

## Why not just write long instructions?
- Custom-agent files load as a **config layer into the spawned session**, so the
  body costs tokens on **every spawn**. Bloat is billed repeatedly.
- GPT-5.6-era models are tuned for **outcome-oriented** prompts. Over-specifying
  the step-by-step procedure adds noise, narrows the search space, and produces
  mechanical output. Describe *what good looks like*, not every keystroke.
- Excessive length **buries** the critical lines. Front-load them.

## The six-section template
```
ROLE: one line — who this agent is and its single job.

SCOPE
- In:  what it should do.
- Out: what it must NOT do (anti-drift — name the adjacent work to avoid).

METHOD
- A few outcome-oriented guidelines: tool preferences (rg/read_file/apply_patch
  over raw shell), parallel reads where independent, priority order. Not a script.

EAGERNESS  (calibrate to role — this is the part most people miss)
- Read/review agents: "stay in lane, don't fix, don't drift."
- Write/worker agents: "bias to action, carry through to verification, make
  reasonable assumptions; stop looping if no progress."

OUTPUT CONTRACT  (the #1 lever for subagents)
- Exactly what to return to the parent, in what shape. The parent only sees what
  the child returns — define it precisely (list, verdict, diff summary, etc.).

DONE
- The completion criteria / stop condition. When is the job finished?
```

## Rules of thumb
- **Enforce hard limits structurally, not in prose.** "Never edit files" belongs
  in `sandbox_mode = "read-only"`; dangerous commands belong in a PreToolUse
  hook. Prose-only rules get ~25–40% compliance; sandbox + hooks are absolute.
- **No preamble/plan chatter in subagents.** They return a result to a parent,
  not to a human watching a rollout. Ask for the contract, not narration —
  upfront-plan prompting can even make the model stop early.
- **Pin model + effort per role.** Cheap tier (Luna) + low effort for read-heavy
  workers; flagship (Sol) + high only for the judgment agent.
- **One job per agent.** If instructions need "also handle X," that's a second
  agent.
- **Size target:** ~12–25 lines of `developer_instructions`. If it's shorter,
  you're probably missing the output contract or the done criteria. If it's much
  longer, you're over-specifying process — move detail into a skill instead.

## When to use a skill vs. a longer agent
If the "how" is a stable, repeatable procedure (release notes, migration steps),
put it in a **SKILL.md** — it loads only when needed (progressive disclosure) and
is portable. Keep the agent body about role, scope, eagerness, and the contract;
let the skill carry the detailed workflow.
