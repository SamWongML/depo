# Global engineering instructions (place at ~/.codex/AGENTS.md)

These are durable rules Codex loads before any project work. Keep this file
small and stable — a steady prefix caches well and cuts token cost. Put
project architecture in the repo's own AGENTS.md, not here.

## Working style
- Make the smallest defensible change. Minimal diffs. Don't touch unrelated files.
- Strict typing, explicit error handling, no silent fallbacks or dead code.
- Prefer editing existing files over creating new ones.
- Explain reasoning in code (docstrings/comments), not in long chat prose.
- Do NOT add "be concise" style rules. GPT-5.6 already writes short output;
  legacy brevity directives now cause it to under-explain.

## Verification (never claim done without this)
- After changes, run the project's build and tests and report exact results.
- If you cannot run validation, state the exact command I should run.
- Lead with concrete findings: file paths, line numbers, reproduction steps.

## Subagents — the important one for GPT-5.6
- Only spawn subagents when I explicitly ask ("spawn", "in parallel",
  "one agent per point"). Do not fan out on your own. 5.6 over-delegates by
  default, and every worker is billed independently.
- When I do ask, describe the split, give each worker a non-overlapping
  deliverable, pass only the context it needs, and wait for all before
  summarizing. One clean summary above, parallel detail below.
- Prefer read-only explorer agents to keep logs, traces, and command output
  OUT of the main thread (prevents context rot).

## Delegation map (when I ask for subagents)
- `explorer`  → read-only code mapping / evidence gathering (cheap model).
- `reviewer`  → correctness, security, missing tests (high-reasoning, read-only).
- `test_writer` → write/repair tests for a bounded target (writes allowed).
- `docs_researcher` → verify framework/API behavior via the docs MCP server.

## Escalation rule (token discipline)
- Start at the lowest model + effort that can produce a checkable artifact.
- If it understood the task but planned/checked poorly → raise EFFORT one step.
- If it misjudged the tradeoff or picked the wrong approach → raise the TIER.
- Never change tier and effort at the same time; you lose the signal about
  which one helped.
